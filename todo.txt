  The honest answer: ~70% eval, ~30% search depth. At only 750 kNPS and depth 10-13, the engine can't calculate its way to strategic truths that
  deeper engines find. But the endgame and zugzwang failures wouldn't improve much with more depth — they need fundamentally better evaluation
  terms.

# Hash Evaluation


  Cache Line Friendliness

  Transposition Table (tt.go) — Well designed

  - TTBucket = 4 slots x 16 bytes = 64 bytes = exactly 1 cache line. This is textbook. A single memory fetch brings the entire bucket into L1.
  - Probe scans all 4 slots sequentially within that one cache line — no additional cache misses.
  - The 4-way set-associative design means collisions (different keys mapping to the same index) have 4 chances before eviction. This is the
  standard approach (Stockfish uses the same layout).

  Pawn Table (pawns.go) — Direct-mapped, collision = eviction

  - PawnEntry = 32 bytes. Direct-mapped (1 entry per index, always-replace).
  - Any collision immediately evicts the old entry. No associativity at all.
  - Two PawnEntry values fit in one cache line, but only one is ever probed — the second is wasted bandwidth.
  - This is the weakest table for collision handling. However, pawn structures repeat heavily (pawn hash space is much smaller than full position
  space), so the hit rate is typically still decent.

  Eval Cache (eval.go) — Direct-mapped, same story

  - EvalEntry = 10 bytes (padded to 16). Direct-mapped, always-replace.
  - 4 entries fit in a cache line but only 1 is probed. Same collision weakness as the pawn table.
  - The eval cache has even higher collision pressure because it's keyed by the full Zobrist hash (huge key space) and is only 1MB by default.

  Collision Impact

  The key question: how much are direct-mapped collisions hurting you?

  For the TT, you're fine — 4-way associativity is the proven approach and you have age-based replacement for stale entries.

  For the pawn table and eval cache, the direct-mapped design means:
  - Two positions that hash to the same index keep thrashing each other
  - In search, this happens frequently because sibling nodes often differ by one non-pawn piece move (same pawn hash → no collision) but the eval
  cache gets hammered by positions that differ by one move mapping to the same slot

  Potential Improvements

  Pawn table — 2-way buckets (low effort, likely win):
  Pack 2 entries per 64-byte cache line. On probe, check both. On store, replace the one that doesn't match (or the older one). This costs zero
  extra cache misses and cuts thrashing significantly. The struct is already 32 bytes, so two entries = 64 bytes = 1 cache line.

  Eval cache — 4-way buckets (low effort, likely win):
  EvalEntry is 16 bytes padded, so 4 fit in a cache line. Make a bucket of 4 entries, probe all 4, always-replace the oldest/first-non-matching.
  Same memory, same cache behavior, 4x fewer collisions.

  Alternatively for eval cache — use a different replacement trick:
  Store only the upper 32 bits of the key (saves space, still catches 99.99999% of collisions) and pack the score alongside it. This shrinks
  entries to 8 bytes, fitting 8 per cache line if you went bucket-style.

  The TT is already optimal for its design. The cheap wins are in making the pawn and eval tables bucket-associative to match what the TT already
  does — same number of cache misses, fewer collisions. Want me to implement either of these?

  - Pawn table: 1 MB / 32 bytes per entry = 32,768 entries (direct-mapped)

  These are quite small. With 65K eval entries, you have only 16 bits of index, so any two positions whose Zobrist hashes share the same lower 16
  bits will collide. In a typical search of millions of nodes, that's a lot of thrashing.

  The pawn table is even smaller at 32K entries, though pawn structures repeat much more than full positions, so the effective collision rate is
  lower.

  For context, the TT defaults to 64 MB in UCI (configurable via Hash option), giving it ~1M buckets with 4-way associativity — orders of
  magnitude more capacity.

  Bumping these to 2-4 MB would be essentially free (memory is cheap) and would directly cut collision rates. Combined with bucket-associativity,
  you'd get a meaningful reduction in redundant eval/pawn computation.

  # Claude Suggestions

  
  Low-Hanging Fruit Optimizations (ranked by effort/reward)

  1. givesCheck after MakeMove — the big one

  At search.go:995, every move that passes legality and isn't pruned pre-MakeMove calls:
  b.MakeMove(move)
  givesCheck := b.InCheck()  // full IsAttacked: 5 piece-type checks + 2 magic lookups

  InCheck() calls IsAttacked() which does 5 conditional checks plus 2 magic bitboard lookups. This runs for every move searched, and it's used to
  gate check extensions, futility pruning, LMP, and LMR.

  The optimization: Compute givesCheck cheaply before MakeMove using the moving piece type and destination square. Most moves don't give check, so
   a fast pre-filter avoids the full IsAttacked call. You only need to check:
  - Does piece X attack the enemy king from the destination square? (1 table lookup)
  - Does the move create a discovered check? (is the piece on a line between own king and enemy king, and moving off that line?)

  Stockfish does this with a checkSquares[pieceType] bitboard precomputed at the start of each node — the set of squares from which each piece
  type attacks the enemy king. Then givesCheck = (checkSquares[piece] & toBB) != 0 || discoveredCheck. This replaces 5 attack checks with 1
  bitboard AND.

  Estimated gain: 5-10% NPS. This is called on every non-pruned move.

  2. Avoid IsLegal make/unmake for en passant

  Currently IsLegal falls through to full make/unmake for en passant moves because of the tricky discovered check case. But en passant is rare
  (~0.1% of moves). This is already well-handled — the cost is negligible. Skip this.

  3. Incremental givesCheck in MakeMove

  A more ambitious version of #1: have MakeMove set a GivesCheck flag on the board. Since MakeMove already knows the piece, from, and to squares,
  it can check discovered + direct check cheaply as part of the move application. This eliminates the separate InCheck() call entirely.

  4. Node time check optimization

  Currently at search.go:656:
  time.Now()
  every 4096 nodes. time.Now() is a vDSO call (~20ns on Linux). Could use a monotonic counter or nanotime directly, but honestly 4096-node batches
   make this negligible. Not worth it.

  5. Move generation isn't allocating in the hot path

  The explorer flagged GenerateAllMoves() as allocating, but that function isn't used in search. The MovePicker uses
  GenerateCapturesAppend/GenerateQuietsAppend/GenerateEvasionsAppend which append to pre-allocated slices (mp.moves). The pickers are
  pre-allocated in info.pickers[ply] and reuse their slices. This is already well-optimized.

  My recommendation

  Do #1 — precomputed check squares. It's the clearest win:
  - Compute checkSquaresBB[6] once per node (6 piece types) using the enemy king square
  - Replace givesCheck := b.InCheck() with givesCheck := isCheckMove(piece, from, to, checkSquares, discoverers)
  - Eliminates a full IsAttacked() per searched move

  This is a well-understood optimization that most strong engines implement. Want me to plan it out?
