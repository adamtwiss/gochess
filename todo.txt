
# Claude's ideas 2nd Feb

  [Done] Lazy SMP (multi-threaded search) -- The engine is single-threaded. On modern hardware this leaves a lot of performance
   on the table. Lazy SMP is the standard approach: multiple threads share the transposition table and search the same
  root position, relying on hash collisions and timing differences for natural work distribution. This is probably the
  single biggest strength gain available.

  Texel tuning -- Every eval parameter (mobility arrays, king safety weights, pawn bonuses, passed pawn values, etc.) is
   hand-tuned. A Texel tuning framework uses a large set of labeled positions (from engine self-play or databases) to
  optimize all constants via gradient descent against game outcomes. This typically finds non-obvious parameter
  interactions that humans miss.

  (in progress - first version failed, tuning mad it netural) Continuation history -- Currently the engine has basic from-to history and counter-moves. Continuation history
  (indexing by the previous 1-2 moves' piece-to-square) significantly improves quiet move ordering, which makes LMR more
   effective and reduces the search tree.

  [Done] SEE pruning of quiet moves in main search -- The engine already uses SEE in quiescence, but applying it to prune
  losing captures and bad quiet moves in the main search (at low depths) is relatively cheap to implement and reduces
  the tree further.

  Syzygy tablebase support -- Perfect endgame play with 6-7 pieces. There are existing Go libraries for probing Syzygy
  tables, so the integration effort is moderate.

  Of the five, Lazy SMP and Texel tuning would give the most visible improvement. Lazy SMP is the more self-contained
  project; Texel tuning requires building a dataset of positions with game results first. What interests you?
