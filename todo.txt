● A few things stood out that don't fit neatly into the NPS optimization
  bucket:

  Correctness / Subtle Bugs

  1. IsLegal for non-king in-check falls back to full MakeMove/UnmakeMove
   (movegen.go:575-579)

  The comment says "rare (~5% of nodes)" but this isn't true — when in
  check, the evasion generator produces fully legal moves and the search
  path uses InitEvasion, which skips IsLegal entirely. The only time this
   inCheck branch in IsLegal executes is when IsLegal is called directly
  with inCheck=true outside the evasion path. Looking at the search, the
  non-evasion path passes inCheck=false at search.go:1062. So this branch
   is effectively dead in the search hot path — it's only reachable via
  IsLegalSlow or external callers. Not a bug, but the comment is
  misleading and the code could be simplified.

  2. EnPassant legality check in IsLegal includes NNUE overhead

  When IsLegal does MakeMove/UnmakeMove for EP validation, it pays the
  full NNUE accumulator push/pop cost. This is wasted work since we only
  need to know if the king is attacked. EP is rare enough that this
  doesn't matter for NPS, but it's an example of the MakeMove path being
  heavier than it needs to be for non-search uses.

  Architectural Observations

  3. Two parallel move generation code paths

  The MoveGen struct (movegen.go:4-44) and its generateAllMoves method
  produce moves via one code path. GenerateAllMovesAppend uses a
  completely different path (GenerateCapturesAppend +
  GenerateQuietsAppend). If a bug is fixed in one, the other won't get
  it. The MoveGen path is only used by GenerateAllMoves(), which is
  called from... ProbCut in search (GenerateCaptures wraps
  GenerateCapturesAppend), tests, and CLI. The old MoveGen struct could
  be deleted entirely and GenerateAllMoves rewritten as a one-liner
  wrapping GenerateAllMovesAppend, eliminating ~300 lines and the
  divergence risk.

  4. The SearchInfo struct serves too many roles

  At 1.4MB, SearchInfo bundles together:
  - Hot per-node data (Nodes, Stopped, Deadline, StaticEvals,
  ExcludedMove, Killers)
  - Warm per-search data (History, CaptHistory, ContHistory)
  - Cold configuration (TT pointer, OnDepth callback, ThreadIndex,
  HelperInfos)
  - Debug stats (LMR/LMP/SEE/Singular counters)
  - Pre-allocated buffers (pickers, pvTable, pvLen)

  Everything lives in one massive struct, so accessing info.Nodes (offset
   0) and info.History (offset 368) and info.ContHistory (offset 18,416)
  all touch different cache lines in the same allocation. The debug stats
   (LMRAttempts etc.) at offset ~1,414,512 are interleaved between
  CaptHistory and ExcludedMove — they're only ever written, never read
  during search, but they force the compiler to keep them in memory.

  Splitting into hot/cold structs (or at minimum, moving the debug
  counters to the end / behind a build tag) would improve spatial
  locality for the fields that actually matter per-node.

  5. helperSearch duplication is a maintenance risk

  The code review agents flagged this too. The aspiration window loop and
   search table clearing are copy-pasted between SearchWithInfo and
  helperSearch. Any future change to aspiration logic, search
  initialization, or time management has to be applied in two places.
  This is the kind of thing that silently diverges over time.

  Strategic Considerations

  6. Endgame scaling is applied inside NNUE eval — may fight the network

  NNUEEvaluateRelative (eval.go:369-398) applies classical endgame scale
  factors (endgameScale()) on top of the NNUE output. The NNUE network
  was trained on positions with game results, so it should already have
  learned drawish patterns like KNN, opposite bishops, etc. Applying
  classical scaling on top may be double-counting drawishness, causing
  the engine to undervalue winning positions that the network correctly
  evaluates as difficult-but-winnable.

  This is a common issue when transitioning from classical to NNUE.
  Stockfish removed most classical scaling when they adopted NNUE. It
  might be worth A/B testing with endgame scaling disabled when NNUE is
  active.

  7. No prefetch for TT probes

  Stockfish issues a __builtin_prefetch for the TT bucket before the
  position is needed — typically right after MakeMove, so the cache line
  is loading while other work happens. Go doesn't have a direct prefetch
  intrinsic, but you can approximate it with an unsafe load-and-discard
  pattern, or via a small assembly stub. Given that TT probes are one of
  the most cache-miss-prone operations (random access into a 16-64MB
  table), prefetching could be worth 2-5% NPS. This would require a small
   assembly function similar to your NNUE SIMD stubs.

  8. The engine doesn't distinguish "root move ordering" from "interior
  node ordering"

  At the root, the cost of trying moves in suboptimal order is amplified
  because every root move triggers a full subtree search. Many engines
  maintain a separate root move list sorted by score from the previous
  iteration, rather than relying on the generic MovePicker. Your
  aspiration window + TT move handles the first move well, but moves 2-N
  at the root could benefit from previous-iteration score sorting. This
  is an Elo improvement more than NPS, but worth noting.

  From the Code Review Agents (worth highlighting)

  9. evaluateKingSafety returns (mg, eg) but eg is always 0

  Similarly evaluateEndgameKings returns (mg, eg) with mg always 0. These
   aren't performance issues but they make the code harder to reason
  about — a reader sees a two-value return and assumes both matter.

  10. abs() defined in movepicker.go but used from search.go

  Minor, but indicative of organic growth without periodic cleanup. The
  function should live somewhere shared, or use Go 1.21+ generics if
  you're on a recent enough version (you're on 1.26+ based on the builtin
   min usage).

  Board Representation & Move Mechanics — NPS Analysis

  1. Magic Struct is 32 bytes — Cache-Unfriendly for Lookups (~2-3% NPS)

  Each Magic struct is 32 bytes (4 × 8-byte fields: Mask, Magic, Shift,
  Offset). The RookMagics[64] and BishopMagics[64] arrays are 2KB each. A
   sliding piece attack lookup (RookAttacksBB at attacks.go:384-388)
  does:

  m := &RookMagics[sq]                              // load 32 bytes
  index := (uint64(occupied & m.Mask) * m.Magic) >> m.Shift  // use 3 of
  4 fields
  return RookAttacks[m.Offset + index]              // use 4th field

  This touches all 32 bytes, but only Mask, Magic, Shift are needed for
  the index computation, and Offset for the final lookup. Since Magic is
  32 bytes, two Magic structs share a cache line — fine for sequential
  access, but during search, square access is essentially random.

  Optimization (Stockfish-style "fancy" magics): Stockfish stores magic
  data as just 3 fields: magic, mask, and a direct pointer to the
  sub-table instead of offset + index. This eliminates the addition and
  the Offset field. More importantly, Shift can be folded into the magic
  number by using "fixed-shift" magics (all rooks use shift=52, all
  bishops use shift=55), eliminating the field entirely. This gets the
  struct to 24 bytes (3 × uint64), with two fitting perfectly in one
  cache line.

  Even more aggressive: the Shift value is constant per piece type in
  many implementations. If you use fixed-shift magics, you can drop Shift
   entirely and hardcode it, bringing the struct to 16 bytes (Mask +
  Magic), with Offset replaced by a pointer stored separately. Four
  structs per cache line.

  2. Castling Rights Update Uses Two Switch Statements (~1% NPS)

  makemove.go:140-163: Every MakeMove executes two switch statements on
  from and to squares, even when 95%+ of moves don't affect castling
  rights. The Go compiler typically compiles this as a series of
  comparisons (not a jump table, since the cases are sparse constants).

  Fix — lookup table: A [64]CastlingRights table indexed by square,
  containing the rights to preserve (mask AND'd with current rights):

  var castlingRightsMask [64]CastlingRights // initialized to AllCastling
   everywhere
  // except: castlingRightsMask[e1] = ^(WhiteKingside|WhiteQueenside),
  etc.

  Then the entire switch block becomes:
  b.Castling &= castlingRightsMask[from] & castlingRightsMask[to]

  One table lookup (always L1 cache — 64 bytes = 1 cache line), two AND
  operations. This is what Stockfish does. Eliminates ~10 branches from
  every MakeMove.

  3. piece.Color() Uses a Branch — Could Be Arithmetic (~0.5-1% NPS)

  piece.Color() at board.go:28-33:
  if p >= BlackPawn { return Black }
  return White

  Called from putPiece, removePiece, movePiece — i.e., multiple times per
   MakeMove/UnmakeMove. Also inlined by the compiler in hot paths. The
  branch is predictable but still executes.

  Alternative: Color((p - 1) / 6) or equivalently Color((p >> 3) & 1) —
  but the piece encoding (1-6 White, 7-12 Black) doesn't map cleanly to
  bit operations. However, a lookup table var pieceColor [13]Color would
  be branch-free and always L1-hot (13 bytes).

  4. movePiece Does 5 Memory Updates + 2 PST Table Lookups

  board.go:221-234: movePiece does:
  - 2 Squares[] writes (from→Empty, to→piece)
  - 1 Pieces[piece] XOR
  - 1 Occupied[color] XOR
  - 1 AllPieces XOR
  - 2 pstCombinedMG lookups + 1 add
  - 2 pstCombinedEG lookups + 1 add

  The PST lookups access pstCombinedMG[piece][sq] — a 6656-byte table (13
   × 64 × 8 bytes). Two random accesses per movePiece call. With piece
  and square changing every call, these touch different cache lines each
  time.

  Observation: The PST lookup is probably the most expensive part of
  movePiece, since Squares, Pieces, Occupied, and AllPieces are all
  within the Board struct's first 4 cache lines (offsets 0-224) and stay
  hot.

  Micro-optimization: Combine MG and EG PST into a single lookup by
  packing both into one int32 (16 bits each, or even using the full 32
  bits for each and packing into int64). This halves the PST cache
  footprint and turns two lookups into one:

  var pstCombined [13][64]int64  // MG in upper 32, EG in lower 32

  Then movePiece does one lookup per square instead of two. Stockfish
  uses a Score type that packs MG+EG into a single int for exactly this
  reason.

  5. MakeMove Zobrist Hashing Does Many Conditional XORs

  makemove.go:74-75: The core hash update is:
  b.HashKey ^= Zobrist.Pieces[piece][from]
  b.HashKey ^= Zobrist.Pieces[piece][to]

  The Zobrist.Pieces table is 6760 bytes (~106 cache lines). With piece
  and squares changing each move, this accesses 2 essentially random
  cache lines per MakeMove. Not avoidable — it's the nature of Zobrist
  hashing.

  But the additional conditional hash updates add branching overhead:
  - Pawn hash (makemove.go:78-81): 2 extra lookups + XORs for pawn moves
  (~25% of moves)
  - En passant (makemove.go:124-136): 2 branches + hash lookup
  - Castling (makemove.go:166-180): 4 conditional XORs only when castling
   changes

  The castling rights hash update is sound. The en passant handling at
  lines 124-136 first removes the old EP hash, then conditionally sets a
  new one — this is correct and necessary.

  Minor optimization: The Zobrist.Pieces[piece][from] ^
  Zobrist.Pieces[piece][to] pattern could be precomputed as
  Zobrist.PieceMoves[piece][from][to], but this would be a 13×64×64×8 =
  416KB table — way too large. Not worthwhile.

  6. UnmakeMove Restores Hash from Undo Stack — This is Already Optimal

  makemove.go:321: b.HashKey = undo.HashKey — direct restore, no re-XOR.
  This is the right approach; some engines re-XOR all the same keys in
  reverse, which is slower. Your implementation is good here.

  7. SquareBB(sq) Uses Lookup Table — Good

  bitboard.go:56-58: squareBBTable[sq] is a 512-byte table (64 × 8 bytes)
   = 8 cache lines. Always L1 hot during search. This is optimal; the
  alternative (1 << sq) involves a variable shift which is typically the
  same speed on modern x86 but slightly slower on some architectures.

  8. Move Generation Append Pattern Creates Slice Overhead

  Throughout movegen.go, moves are appended one at a time:
  moves = append(moves, NewMove(from, to))

  Go's append has overhead per call: it checks capacity, potentially
  grows the backing array, updates the slice header. With ~30-50 moves
  per position and the slice pre-allocated to sufficient capacity (so no
  reallocation), the per-append cost is small but non-zero — a bounds
  check + length increment + write.

  Alternative (Stockfish-style): Use a Move* pointer (or in Go, an index)
   and advance it directly:
  moves[idx] = NewMove(from, to)
  idx++

  This eliminates the append bookkeeping. The challenge in Go is that
  slices are the idiomatic way, and the compiler may not optimize append
  to a simple index increment even when capacity is guaranteed.

  Practical approach: Pre-size the slice to a known maximum (e.g., 218 —
  theoretical max legal moves), use index-based writes, then slice to
  actual count at the end. This would be a micro-optimization with ~1%
  impact at most, but it applies to every single movegen call.

  9. pieceOf Lookup Table is Well-Designed

  board.go:37-51: pieceOfTable[p][c] is a 7×2 = 14 entries, 14 bytes.
  Always L1 hot. Clean and fast. No issue here.

  10. IsAttacked Early-Exit Order is Good

  movegen.go:409-435: Checks pawns → knights → king → bishops/queens →
  rooks/queens. This order prioritizes cheaper lookups (direct table
  reads) before expensive ones (magic bitboard multiplies). The pawn
  check is first since pawns are the most common pieces and pawn attacks
  are a single table lookup.

  11. Board.Squares[64] as Piece (int8) is Space-Efficient but Forces
  Type Conversions

  Squares is [64]Piece where Piece = int8. The array is 64 bytes = 1
  cache line. Every access returns int8 which must be widened for array
  indexing (Zobrist, PST, etc.). Go does this implicitly, but it adds
  sign-extension instructions.

  If Piece were uint8 instead of int8, the sign extension would be
  eliminated. However, Empty = 0 and pieces are 1-12, so there's no sign
  issue in practice — the compiler should optimize this away. Low
  priority.

  12. NewSquare(file, rank) in Castling Code is Computed at Call Time

  makemove.go:101-119: Castling code calls NewSquare(6, 0), NewSquare(7,
  0), etc. repeatedly. These are rank*8 + file = simple arithmetic. The
  Go compiler should constant-fold these, but it's worth verifying. If
  not constant-folded, replacing with named constants (const G1 =
  Square(6), const F1 = Square(5), etc.) would ensure zero computation.

  Search Speed Optimisations

    Search NPS Analysis

  1. TT Probe/Store Atomic Stats are Pure Overhead (~2-3% NPS)

  tt.go:126: atomic.AddUint64(&tt.probes, 1) is called on every TT probe
  — once per negamax node + once per qsearch node. That's 100% of nodes
  paying an atomic increment. Then atomic.AddUint64(&tt.hits, 1) on hits
  (~60-70% of probes), and atomic.AddUint64(&tt.writes, 1) on stores.

  Atomic adds are ~5-15ns each on x86 (LOCK XADD forces cache line
  ownership). At 1M NPS, that's 3 atomics × ~10ns × 1M = ~30ms/s, i.e.
  ~3% overhead for stats nobody reads during search. Stats() is only
  called from UCI info strings.

  Fix: Remove the stats atomics from the hot path entirely. If you need
  TT fill stats, compute Hashfull() on demand (which you already do by
  sampling). Or count probes/hits as regular uint64 fields on SearchInfo
  (per-thread, no atomics needed).

  2. time.Now() Every 1024 Nodes (~1-2% NPS)

  search.go:759-763 and search.go:1529-1534: time.Now().UnixNano() is
  called every 1024 nodes in both negamax and qsearch. time.Now() in Go
  involves a vDSO call (~20-30ns). At 1M NPS, that's ~1000 calls/sec ×
  25ns = 25µs/sec — small individually, but this is compounded by the
  fact that you check in both negamax and qsearch, so you're checking
  roughly every 512 effective nodes.

  More importantly, the node counter check itself (info.Nodes & 1023)
  runs every single node. With the stopped check right after
  (atomic.LoadInt32(&info.Stopped) at lines 766 and 1536), that's an
  atomic load per node even when the timer isn't firing.

  Fix: Increase the check interval to 4096 (your CLAUDE.md already says
  4096 but the code says 1023). Also, the atomic.LoadInt32(&info.Stopped)
   on lines 766/1536 could be gated inside the Nodes&mask == 0 block
  rather than checked independently every node — if Stopped is set by the
   timer check or by another thread's stop, a few hundred extra nodes
  before noticing won't matter.

  3. SearchInfo Struct is 1.4MB — ContHistory Dominates (~2-4% NPS)

  ContHistory [13][64][13][64]int16 is 1,384,448 bytes (1.32MB) — 96% of
  the SearchInfo struct. This is accessed per-node for quiet move scoring
   and LMR adjustment (search.go:1321-1323). But the access pattern is:

  contHistPtr = &info.ContHistory[prevPiece][prevTo]  // select sub-table
   (once per node)
  score += int(contHistPtr[piece][move.To()])           // per-move
  lookup

  The inner table is [13][64]int16 = 1,664 bytes = ~26 cache lines. The
  outer indexing [prevPiece][prevTo] means you're jumping to a different
  1.6KB sub-table every node (because prevPiece+prevTo changes). With
  1.32MB total, this thrashes L1 (32KB) and puts pressure on L2 (256KB).

  This is inherent to the data structure — Stockfish has the same issue.
  But there's a subtle implementation difference: Stockfish stores
  continuation history as a pointer to the sub-table on the stack,
  resolved once per node. You do the same (contHistPtr), so the access
  pattern is sound. However, the zero-initialization at search start
  (search.go:279: info.ContHistory = [13][64][13][64]int16{}) writes
  1.32MB of zeros, evicting hot cache lines. Consider memclr via for
  range or just not clearing (Stockfish ages history between searches
  rather than clearing).

  Opportunity: Consider halving the table to [7][64][13][64]int16 (index
  by piece type 1-6 instead of piece 1-12 — White and Black pieces of the
   same type have the same tactical meaning). This halves the table to
  ~660KB, improving cache utilization. Stockfish uses piece type indexing
   for exactly this reason.

  4. CheckData Called Unconditionally (~1-2% NPS)

  search.go:874: checkSq, discoverers := b.CheckData(b.SideToMove) runs
  at every non-leaf, non-check node. It computes:
  - 5 attack bitboard lookups (pawn, knight, bishop, rook, queen to enemy
   king)
  - A sniper loop (BishopAttacks + RookAttacks with empty board, then
  BetweenBB scan)

  This is ~50-100ns per call. But the data is only consumed after
  MakeMove at search.go:1165-1178, and only for non-special-flag moves.
  Nodes that get pruned before reaching the move loop (NMP, RFP,
  razoring, ProbCut) pay the CheckData cost for nothing.

  Fix: Compute checkSq/discoverers lazily — either defer to first use, or
   compute it after the pruning section (move it to just before line
  1030).

  5. ProbCut Allocates + Calls IsLegalSlow (~0.5-1% NPS)

  search.go:984: b.GenerateCaptures() allocates a fresh []Move slice.
  Then at line 993, b.IsLegalSlow(m) calls IsPseudoLegal +
  PinnedAndCheckers per move. But PinnedAndCheckers was already computed
  at line 872! And ProbCut only runs when !inCheck, so the pinned
  bitboard from line 872 is valid.

  Fix: Use a pre-allocated buffer (stack [64]Move array) and replace
  IsLegalSlow with IsLegal(m, pinned, false) using the already-computed
  pinned.

  6. IsLegal for En Passant Does Full MakeMove/UnmakeMove

  movegen.go:559-563: When flags == FlagEnPassant, IsLegal falls back to
  full MakeMove + IsAttacked + UnmakeMove. This is correct (EP has rare
  discovered check edge cases), but MakeMove includes the full NNUE
  accumulator push/pop, undo stack append, hash update, etc. — all wasted
   work since we only need to check if the king is attacked.

  EP is rare (~1-2% of moves), so this is maybe 0.1% NPS overall. But if
  you wanted to optimize it: a lightweight "EP legality check" that
  temporarily removes both the moving pawn and captured pawn from the
  occupancy bitboard, then checks if the king is attacked by sliders
  along that rank, would be much cheaper.

  7. Board Struct Layout is Decent but Could Be Tighter

  Board is 304 bytes = ~5 cache lines. The hot fields in the inner search
   loop are:
  - Squares[64] (offset 0, 64 bytes) — accessed for every move's piece
  lookup
  - SideToMove (offset 64) — checked every node
  - Pieces[13] (offset 96, 104 bytes) — accessed for attack generation
  - Occupied[2] (offset 200, 16 bytes) — accessed for attack gen
  - AllPieces (offset 216, 8 bytes) — accessed for magic bitboard lookups
  - HashKey (offset 80) — accessed for TT probe/store

  The first 3 cache lines (0-191) cover Squares, SideToMove, Castling,
  EnPassant, HalfmoveClock, FullmoveNum, HashKey, PawnHashKey, and
  Pieces[0..11]. This is reasonably good — the hottest fields are all
  within 4 cache lines.

  One layout concern: HashKey at offset 80 shares a cache line with
  HalfmoveClock and FullmoveNum (offsets 68-79), which is fine. But
  Pieces[13] at offset 96 starts on a new cache line (line 1: 64-127),
  which means accessing Pieces[WhiteKnight] through Pieces[WhiteKing]
  (indices 2-6, offsets 112-144) spans cache lines 1-2. Not easily
  fixable without a major restructure.

  8. UndoStack Append in MakeMove

  makemove.go:44: b.UndoStack = append(b.UndoStack, undo) copies 24 bytes
   of UndoInfo onto the slice. The slice is pre-allocated to 256
  capacity, so no allocations, but the append still involves Go runtime
  overhead (length check, update slice header). At ~2M MakeMove calls per
   second, this adds up.

  The UndoInfo is already well-packed at 24 bytes. No obvious improvement
   here without moving to a fixed-size stack array, which would remove
  the slice overhead but limit max depth.

  9. IsRepetition Scans Undo Stack Every Node

  board.go:597-609: Called at every node with ply > 0. Scans backward
  through the undo stack by 2 (checking every other position). In the
  worst case (long game with HalfmoveClock = 100), this scans 50 entries.
   The access pattern is cache-friendly (sequential backward), but the
  sheer frequency (every node) makes this significant.

  Opportunity: Stockfish uses a compact hash history array instead of
  scanning. Store recent position hashes in a fixed-size ring buffer
  indexed by ply. Repetition check becomes a simple loop over a
  contiguous array of uint64s rather than reaching into UndoInfo structs
  (which have 24-byte stride, so you're skipping 16 bytes of non-hash
  data per iteration).

  10. Dual Stopped Checks Per Node

  search.go:759-768: Every negamax node does:
  if info.Nodes&1023 == 0 {    // branch 1
      if time check ...         // rare
  }
  if atomic.LoadInt32(&info.Stopped) != 0 {  // branch 2 — EVERY node
      return 0
  }

  The atomic.LoadInt32 on line 766 runs unconditionally every single
  node. With single-threaded search, this is just a regular memory load
  (no contention), but it's still a branch that the branch predictor must
   track. In multi-threaded mode, this becomes a real atomic load with
  cache-line ping-pong potential if multiple threads are checking.

  Fix: Move the Stopped check inside the Nodes&mask block. The search
  will run at most 1023 extra nodes before stopping — completely
  unnoticeable.

  Move Ordering Analysis

  Issues with Current Implementation

  1. Qsearch doesn't use TT move for ordering (HIGH impact)

  search.go:1655 — InitQuiescence(b, &info.CaptHistory) doesn't pass the
  ttMove. The TT move is probed (search.go:1548) but never used for move
  ordering. In most engines, trying the TT move first in qsearch is a
  significant win — it's the move from a previous deeper search that was
  best, so it often produces an instant cutoff without generating/scoring
   all captures.

  Fix: Add a InitQuiescenceWithTT(b, ttMove, captHist) variant or modify
  InitQuiescence to accept an optional TT move and start at stageTTMove
  when one exists.

  2. ProbCut allocates a fresh slice every node (search.go:984)

  b.GenerateCaptures() calls make([]Move, 0, 32) internally. ProbCut runs
   at every node with depth >= 5 where the eval condition is met. This
  creates GC pressure. Should use a pre-allocated buffer (e.g. a
  stack-local [64]Move array with slice header, or a pre-allocated field
  on SearchInfo).

  3. SEE called for every capture during scoring (movepicker.go:237)

  generateAndScoreCaptures calls b.SEE(m) for every capture to partition
  into good/bad. Full SEE is expensive (~100-200ns per call due to
  leastValuableAttacker loop). Stockfish uses SEESign(m, threshold) with
  early exits instead of full SEE for partitioning. You already have
  SEESign — the partition only needs to know if SEE >= 0, so use
  SEESign(m, 0) instead of SEE(m).

  For bad captures, you'd lose the exact SEE value for scoring, but bad
  captures are the lowest priority stage anyway — their exact ordering
  rarely matters. You could score bad captures by -MVV-LVA + captHist
  instead.

  Estimated impact: ~5-10% reduction in time spent on capture scoring.

  4. Selection sort is O(n^2) worst case, scans full remaining list each
  call

  pickBest() (movepicker.go:331-355) scans from mp.index to len(mp.moves)
   every time. For quiet moves (typically 30-50 per position), this means
   the first call scans ~40 moves, second scans ~39, etc. With beta
  cutoffs typically happening in the first 1-5 moves, most scans are
  wasted. This is the standard approach (Stockfish does the same), but...

  Opportunity: For the quiet move stage specifically, if there are many
  moves and cutoff typically happens early, the selection sort is fine.
  But consider: if you have >30 quiets and LMP will prune after 3 +
  depth*depth (e.g., 7 at depth 2), you're scoring and scanning 30+ moves
   but only ever using 7. Lazy generation of quiet scores could help —
  only score the next batch when needed.

  5. IsPseudoLegal is called redundantly for killers/counter-move

  At movepicker.go:150,158,169, killer and counter-move validation calls
  IsPseudoLegal, which does a full piece type switch including pawn
  direction checks, castling validation, etc. This is ~20-30 array
  lookups per call. For killers and counter-move (3 calls per node), this
   adds up.

  Opportunity: A lighter validation could be used for killers — e.g.,
  just check the from-square has the right piece and the to-square isn't
  our piece. The full movegen validation is only needed for exotic cases
  (castling, en passant, double pawn push). Since killers were legal
  moves from sibling nodes, a lighter check suffices.

  6. isCapture check for killers/counter-move is redundant

  movepicker.go:150,158,169 — After IsPseudoLegal, there's an
  !mp.isCapture() check. Killers should never be captures (they're stored
   only for quiet moves on beta cutoff). The counter-move similarly
  should be quiet. If you enforce this at store time (which you do —
  search.go:1412 only stores quiet cutoff moves as killers), the capture
  check is wasted work.

  However: The IsPseudoLegal + isCapture overhead is only 3 calls per
  node, so the absolute impact is small.

  Missing Optimizations Found in Top Engines

  7. No TT move in qsearch evasions

  search.go:1583 — InitEvasion in qsearch passes ttMove, which is good.
  But looking more carefully, it passes nil for history and contHist, so
  evasion quiet moves in qsearch have no history-based ordering. This is
  probably fine since qsearch evasions are rare.

  8. No "countermove history" (second continuation history table)

  Stockfish uses two continuation history tables: one for the move 1 ply
  ago, another for the move 2 plies ago. Your engine uses one
  (contHistPtr from the immediate parent). Adding a second contHist from
  2 plies ago is a proven ~5-10 Elo gain in Stockfish. The memory cost is
   another ~1.3MB per thread, and the scoring/update code is nearly
  identical to what you already have.

  9. No "pawn history" table

  Modern Stockfish (since ~2023) uses
  PawnHistory[pawnStructureHash][piece][to] which captures positional
  patterns tied to the pawn structure. This is a significant Elo
  improvement in Stockfish. It's more complex to implement but could be
  worth exploring.

  10. CheckData called unconditionally (search.go:874)

  checkSq, discoverers := b.CheckData(b.SideToMove) is called at every
  non-leaf node, even when in check (where it's not used — evasion mode
  doesn't need check detection for child moves). When inCheck, this work
  is wasted since the evasion path doesn't use checkSq/discoverers.

  Fix: Move the CheckData call inside the !inCheck branch, or defer it
  until the move loop actually needs it (after MakeMove). The data is
  only used at search.go:1165-1178 for post-MakeMove check detection.

  11. No SEE-based ordering in qsearch captures

  InitQuiescence uses stageGenerateCaptures directly, which calls
  generateAndScoreCaptures(). This runs full SEE for partitioning. But in
   qsearch, you already call SEESign(move, 0) at line 1678 to filter bad
  captures. This means captures are SEE-evaluated twice: once during
  scoring (full SEE), and again during the search loop (SEESign).

  Fix: In qsearch, skip the SEE partition and just score captures by
  MVV-LVA+captHist, then use the existing SEESign call in the search loop
   as the filter. This avoids the double-SEE cost.

NNUE 7th March

  Model topology changes (wider accumulator, king buckets) are relatively easy to implement — we change some constants and retrain. The code
  infrastructure is already there. Going to 512 accumulators doubles the feature transformer capacity and typically gives +50-100 Elo in other
  engines. King buckets (e.g., mirroring the board so king is always on files a-d, giving 32 buckets instead of 64 king squares) reduce the
  feature space while making the features more meaningful.

  Input feature changes are a different beast. Our current HalfKP uses:
  king_square × piece_type × piece_square = 64 × 10 × 64 = 40960 features

  What ChatGPT is describing — threat features, pawn structure motifs, attack maps — is essentially trying to hand-engineer what the accumulator
  already learns automatically. The whole point of the NNUE accumulator is that it discovers these patterns from data. Adding explicit threat
  features would be like telling the network "here's a hint about what to look for."

  The evidence from Stockfish's evolution is telling:
  - They went from HalfKP → HalfKA (added king-king info) — modest gain
  - Then to HalfKAv2 with king buckets — bigger gain
  - Then wider accumulators (256→512→1024) — consistent gains
  - They never added explicit threat/mobility/pawn-structure input features

  The reason is that the accumulator naturally learns these patterns, and adding explicit features adds complexity without giving the network
  information it couldn't discover on its own. The accumulator neurons we discussed earlier — the ones that specialize into "knight outpost
  detector" or "king attack pattern" — those are the threat and pawn structure features, just learned rather than engineered.

  My ranking for our engine:

  1. More/better training data — free Elo, no code changes
  2. Wider accumulator (512) — straightforward, proven gains
  3. King buckets — reduces feature space, helps the network generalize across mirrored positions
  4. HalfKA — adds king-to-king relationship
  5. Explicit tactical features — probably not worth it given the above

  The topology changes (2-4) are where the real architectural gains are. The explicit features approach is more of a "classical eval thinking
  applied to neural nets" — it works but it's swimming against the current of what makes NNUE powerful in the first place.

  ---
  Regarding your point about search tuning — you're absolutely right. The search parameters were tuned ~200 Elo ago against classical eval,
  largely using benchmark scores as the signal. Now that:

  1. NNUE changes the search landscape — different eval characteristics (more continuous, potentially less extreme in some positions) mean the
  optimal pruning thresholds, reduction formulas, and extension criteria may be different
  2. Self-play is a better signal — benchmark scores test if you find the right move, but self-play tests if you actually win more games. A
  parameter change could find fewer "correct" moves on benchmarks but play stronger overall

  The right order is:
  1. Get NNUE as strong as possible first — more data, potentially architectural improvements (HalfKA, king buckets), self-play training loop
  2. Then re-tune search parameters with NNUE active — using self-play Elo as the fitness function

  Some search parameters that are likely to shift with NNUE:
  - Futility/RFP margins — NNUE scores may have different centipawn calibration than classical eval
  - LMR reductions — if NNUE eval is more accurate, you can prune more aggressively (or less, if it's smoother)
  - Null move verification depth — NNUE may handle zugzwang differently
  - Passed pawn extensions — as we discussed, NNUE may need more search depth to resolve endgame patterns that classical eval handles statically
  - SEE thresholds — if NNUE piece values differ from the hardcoded SEE values

  The self-play approach for search tuning would be something like SPRT testing individual parameter changes — pick a parameter, try a few values,
   run 1000+ game matches to get statistically significant Elo differences. Slower than benchmark comparison but much more reliable.

Claude March 6th

  1. Time management fixes (high impact, low effort)

  You're still losing games on time in gauntlets — that's free Elo thrown
   away. The engine already has soft/hard deadlines and stability
  scaling, but the MoveOverhead handling and time allocation for short

  2. Syzygy tablebases (high impact, moderate effort)

  Your endgame is the weakest area by far (8% on Eigenmann endgame
  positions). Tablebases give perfect play with 6-7 pieces and eliminate
  entire classes of drawn/lost endgames the engine currently misplays. Go
   libraries exist for probing. This also fixes the "exchanging down to
  fortress positions" problem you noted.

  3. NPS / speed (moderate impact, moderate effort)

  You're 50-100% slower than comparable engines. The biggest single win
  identified was precomputed check squares — replacing the full InCheck()
   call after every MakeMove with a single bitboard AND. Estimated 5-10%
  NPS gain. The pawn/eval table bucket-associativity improvements (2-way

Claude's latest ideas (March 3rd)

 Search Review: Findings & Recommendations

  Based on a detailed comparison of our implementation against Stockfish (SF18 era), Ethereal, Weiss, and Berserk.
  Organized by estimated impact.

  ---
  HIGH IMPACT

  1. History Gravity Formula (missing)

  Current: We use depth * depth for both bonus and malus, clamped to ±32000. Values accumulate without natural decay — a
   move that gets a big bonus early stays high even as it becomes stale.

  Best practice: All top engines use the "gravity" formula:
  newValue = oldValue + bonus - oldValue * abs(bonus) / D
  This naturally decays old values when new evidence arrives. Stockfish uses different D constants per table type (7183
  for butterfly, 10692 for capture history, 30000 for continuation). Ethereal uses 16384 uniformly. Weiss uses per-type
  divisors (4373-16384).

  Also: Stockfish uses asymmetric bonus/malus — the penalty for moves tried before cutoff is ~1.5x larger than the bonus
   for the cutoff move itself (stat_malus > stat_bonus). We use the same depth*depth for both.

  Recommendation: Implement gravity formula. Start with a uniform D=16384. Then consider asymmetric bonus (linear
  min(251*depth-267, 2418)) / malus (linear min(532*depth-163, 693)). This affects move ordering quality everywhere —
  LMR accuracy, killer relevance, quiet scoring.

  2. Continuation History Depth (1-ply → 2+ plies)

  Current: We track only 1-ply-back continuation history (ContHistory[prevPiece][prevTo][curPiece][curTo]).

  Best practice: Ethereal tracks 2 plies (1+2). Weiss tracks 3 (1+2+4). Berserk tracks 4 (1+2+4+6). Stockfish tracks 6
  with tuned per-ply weights (1106, 705, 316, 572, 126, 427).

  The 2-ply-back "follow-up history" captures "after I played move X, two plies later move Y tends to be good" — this is
   the natural response pattern (my move, opponent's reply, my follow-up). It's consistently the second-highest-weighted
   table across all engines.

  Recommendation: Add 2-ply-back continuation history first. This is the highest-value addition. Then consider 4-ply if
  testing shows gains.

  3. Singular Extensions: Depth Gate Too High (10 → 6-8)

  Current: depth >= 10, margin ttScore - depth*3.

  Best practice: Stockfish uses depth >= 6, Berserk depth >= 6, Weiss depth > 4, Ethereal depth >= 8. Our gate of 10
  means we miss singular extensions at depths 6-9, which is a significant portion of the search tree.

  Also: Our negative extensions are only -1. Stockfish does -2 to -3. Berserk does -2 at cut nodes. More aggressive
  negative extensions (multicut) prune large subtrees when alternatives are strong.

  Recommendation: Lower to depth >= 7 or depth >= 8 and test. Consider increasing negative extension to -2 at cut nodes.

  4. Reverse Futility Pruning: Depth Limit Too Low (6 → 8-10)

  Current: depth <= 6, margin depth * 120.

  Best practice: Stockfish goes up to depth < 16. Berserk uses depth <= 10. Ethereal uses depth <= 8. Our margin of
  depth * 120 is reasonable for shallow depths, but we're leaving pruning on the table at depths 7-10.

  Also: Stockfish scales the margin by improving (reduces margin when improving, making it more conservative). Weiss
  adjusts by opponent's history score. Berserk adds opponentDeclining bonus.

  Recommendation: Raise to depth <= 8 first, test. Consider making margin improving-aware: e.g., depth * 120 - 60 *
  improving.

  5. Aspiration Windows: Too Wide (25 → 8-12)

  Current: Initial delta of 25. Widening: 25→100→500→Infinity (discrete jumps).

  Best practice: Stockfish starts at 5 (+ thread diversification). Berserk uses 8. Ethereal uses 10. Weiss uses 9 +
  prevScore²/16384. All use multiplicative widening (1.25x-1.5x per fail) rather than discrete jumps.

  Smaller initial windows mean more positions resolve exactly on the first try, saving the overhead of re-searches.
  Gradual widening avoids the cost of jumping straight to a wide window.

  Recommendation: Reduce to delta=12, widen with delta += delta/2 (1.5x per fail, like Ethereal). This is low-risk,
  high-frequency improvement.

  ---
  MEDIUM IMPACT

  6. LMR: Captures Not Reduced

  Current: We skip LMR entirely for captures. Only quiet moves are reduced.

  Best practice: All four reference engines reduce captures. Ethereal uses a separate formula (R = 3 - captHist/4952).
  Weiss has a separate, gentler LMR table for captures. Berserk reduces non-PV captures. Stockfish runs captures through
   the same unified LMR pipeline.

  Bad captures (low capture history) searched late in the move list are wasting full-depth effort.

  Recommendation: Add capture LMR with a conservative approach: only reduce captures with bad capture history, at non-PV
   nodes, with a smaller base reduction than quiets.

  7. LMR: Missing Adjustment Terms

  Current: We adjust for: PV (-1), cut node (+1), improving (-1), history (histScore/5000). That's 4 terms.

  Best practice additions worth considering:
  - TT move present (-1): When we have a TT move, the position is better understood, so reduce other moves more
  cautiously. Stockfish weights this heavily (-2069 in fractional).
  - ALL-node detection (+1): At expected all-nodes (not PV, not cut), moves are likely to fail low — reduce more
  aggressively.
  - Killer/counter exemption: We already exempt killers from LMR entirely. Some engines instead reduce them less
  (Berserk: -2 for killers/counters). Our current approach (full exemption) may be too generous.

  Also: Our history divisor of 5000 is in the right ballpark (Ethereal uses 6167, Weiss 8870, Berserk ~8192). Could be
  worth tuning.

  8. Null Move: Only at Cut Nodes

  Current: We try null move at any non-root, non-check node with non-pawn material.

  Best practice: Stockfish only does null move at cut nodes. The reasoning: null move is testing "is my position so good
   that even passing gives me a beta cutoff?" This is the definition of a cut node. At ALL-nodes, null move wastes time
  because the position is likely failing low anyway.

  Also: Our reduction R = 3 + depth/3 + min(3, (eval-beta)/200) is less aggressive than Stockfish's R = 7 + depth/3. We
  could increase the base.

  Recommendation: Gate null move on cut nodes (or at least test it). Increase base R to 4 or 5.

  9. IIR: Depth Gate Too Low (5 → 6)

  Current: depth >= 5 when no TT move and not in check.

  Best practice: Stockfish uses depth >= 6 with an additional guard on ALL-node status. Applying IIR too aggressively at
   shallow depths can hurt because you're reducing already-short searches.

  Recommendation: Raise to depth >= 6. Minor but easy to test.

  10. Futility Pruning: Margin Too Large

  Current: staticEval + depth*200 <= alpha. At depth 6, that's a 1200cp margin — extremely wide. This means we almost
  never prune.

  Best practice: Stockfish uses lmrDepth * 120 (uses post-LMR depth, not raw depth). Ethereal uses 77 + lmrDepth * 52.
  Berserk uses 66 + 44 * lmrDepth.

  The key insight is using lmrDepth (depth after LMR reduction) rather than raw depth. Most moves at depth 6 will be
  reduced to depth 2-3 by LMR, so the effective futility margin should be based on that reduced depth.

  Recommendation: Switch to lmrDepth-based margin. Start with 100 + lmrDepth * 100 or similar.

  11. SEE Quiet Pruning: Linear → Quadratic Margin

  Current: -depth * 80 (linear). At depth 8: -640cp.

  Best practice: Stockfish uses -25 * lmrDepth² (quadratic). Berserk uses -13.8 * lmrDepth². Ethereal uses -64 * depth
  (linear but tighter).

  Quadratic scaling makes shallow pruning tighter (more aggressive) while keeping deep pruning conservative. At
  lmrDepth=2: -100cp (Stockfish) vs -160cp (ours). At lmrDepth=5: -625cp vs -400cp.

  Recommendation: Switch to quadratic: -20 * depth * depth or similar.

  ---
  LOWER IMPACT (but easy wins)

  12. Node Check Frequency (4096 → 1024)

  Current: Check clock every 4096 nodes.

  Best practice: Stockfish checks every 512. At fast time controls, 4096 nodes can be several milliseconds — enough to
  significantly overshoot time limits.

  Recommendation: Reduce to 1024. Nearly free change.

  13. LMP: Formula vs Table

  Current: Hardcoded table {0, 5, 8, 12, 18, 25, 34, 44, 56}.

  Best practice: Stockfish uses (3 + depth²) / (2 - improving). Cleaner, extends to any depth, and inherently scales by
  improving status.

  Recommendation: Replace table with formula. Our table values are close to Stockfish's not-improving column already.

  14. Time Management: More Factors

  Current: Best-move stability (0.5x-1.0x) and score delta (up to 1.4x).

  Best practice: Stockfish uses 4 multiplicative factors: falling eval, move stability (sigmoid curve), best-move
  instability (change frequency), and effort concentration (one dominant move → reduce time).

  The "effort concentration" factor is interesting and easy to add: if >90% of nodes are spent on the best move, we're
  very confident and can stop early.

  Recommendation: Add effort-based time saving. Track nodes spent on best root move vs total.

  ---
  FUTURE / LARGER PROJECTS

  15. Correction History (not yet, but on the radar)

  This is the biggest technique we're missing from the 2024-2025 era. Stockfish has 5 types (pawn, minor, non-pawn per
  color, continuation). Weiss has extensive correction history. Berserk has pawn + continuation.

  The concept: when search reveals the true value differs from static eval, store the error indexed by position features
   (pawn hash, piece config). Future positions with similar features get their eval adjusted.

  This is a significant implementation effort (new data structures, update logic, integration into eval and pruning
  margins). Worth doing but probably after the above items are addressed.

# elo

##  Rules of thumb I find handy:

  - 100 Elo ≈ 2:1 odds — the most useful anchor point
  - 400 Elo ≈ 10:1 odds — a full piece advantage in "engine class"
  - 200 Elo ≈ 3:1 — a clearly dominant engine but the weaker one still wins occasionally
  - Every 70-80 Elo roughly doubles the win:loss ratio (ignoring draws)
  - Doubling time control is worth roughly 50-70 Elo for most engines
  - Doubling hash size is worth roughly 5-15 Elo (diminishing returns)

## Performance to ELO

Elo ≈ 70 * log2(speedup)

So a 2x increase in NPS is only worth 70 ELO.
A 10% performance increase is worth 10 ELO.

Given more data, and some Texel/NNUE training has often found us 50+ ELO, that's like a 70% speed increase.

# Hash Evaluation


  Cache Line Friendliness

  Transposition Table (tt.go) — Well designed

  - TTBucket = 4 slots x 16 bytes = 64 bytes = exactly 1 cache line. This is textbook. A single memory fetch brings the entire bucket into L1.
  - Probe scans all 4 slots sequentially within that one cache line — no additional cache misses.
  - The 4-way set-associative design means collisions (different keys mapping to the same index) have 4 chances before eviction. This is the
  standard approach (Stockfish uses the same layout).

  Pawn Table (pawns.go) — Direct-mapped, collision = eviction

  - PawnEntry = 32 bytes. Direct-mapped (1 entry per index, always-replace).
  - Any collision immediately evicts the old entry. No associativity at all.
  - Two PawnEntry values fit in one cache line, but only one is ever probed — the second is wasted bandwidth.
  - This is the weakest table for collision handling. However, pawn structures repeat heavily (pawn hash space is much smaller than full position
  space), so the hit rate is typically still decent.

  Eval Cache (eval.go) — Direct-mapped, same story

  - EvalEntry = 10 bytes (padded to 16). Direct-mapped, always-replace.
  - 4 entries fit in a cache line but only 1 is probed. Same collision weakness as the pawn table.
  - The eval cache has even higher collision pressure because it's keyed by the full Zobrist hash (huge key space) and is only 1MB by default.

  Collision Impact

  The key question: how much are direct-mapped collisions hurting you?

  For the TT, you're fine — 4-way associativity is the proven approach and you have age-based replacement for stale entries.

  For the pawn table and eval cache, the direct-mapped design means:
  - Two positions that hash to the same index keep thrashing each other
  - In search, this happens frequently because sibling nodes often differ by one non-pawn piece move (same pawn hash → no collision) but the eval
  cache gets hammered by positions that differ by one move mapping to the same slot

  Potential Improvements

  Pawn table — 2-way buckets (low effort, likely win):
  Pack 2 entries per 64-byte cache line. On probe, check both. On store, replace the one that doesn't match (or the older one). This costs zero
  extra cache misses and cuts thrashing significantly. The struct is already 32 bytes, so two entries = 64 bytes = 1 cache line.

  Eval cache — 4-way buckets (low effort, likely win):
  EvalEntry is 16 bytes padded, so 4 fit in a cache line. Make a bucket of 4 entries, probe all 4, always-replace the oldest/first-non-matching.
  Same memory, same cache behavior, 4x fewer collisions.

  Alternatively for eval cache — use a different replacement trick:
  Store only the upper 32 bits of the key (saves space, still catches 99.99999% of collisions) and pack the score alongside it. This shrinks
  entries to 8 bytes, fitting 8 per cache line if you went bucket-style.

  The TT is already optimal for its design. The cheap wins are in making the pawn and eval tables bucket-associative to match what the TT already
  does — same number of cache misses, fewer collisions. Want me to implement either of these?

  # Claude Suggestions

  
  Low-Hanging Fruit Optimizations (ranked by effort/reward)

  1. givesCheck after MakeMove — the big one

  At search.go:995, every move that passes legality and isn't pruned pre-MakeMove calls:
  b.MakeMove(move)
  givesCheck := b.InCheck()  // full IsAttacked: 5 piece-type checks + 2 magic lookups

  InCheck() calls IsAttacked() which does 5 conditional checks plus 2 magic bitboard lookups. This runs for every move searched, and it's used to
  gate check extensions, futility pruning, LMP, and LMR.

  The optimization: Compute givesCheck cheaply before MakeMove using the moving piece type and destination square. Most moves don't give check, so
   a fast pre-filter avoids the full IsAttacked call. You only need to check:
  - Does piece X attack the enemy king from the destination square? (1 table lookup)
  - Does the move create a discovered check? (is the piece on a line between own king and enemy king, and moving off that line?)

  Stockfish does this with a checkSquares[pieceType] bitboard precomputed at the start of each node — the set of squares from which each piece
  type attacks the enemy king. Then givesCheck = (checkSquares[piece] & toBB) != 0 || discoveredCheck. This replaces 5 attack checks with 1
  bitboard AND.

  Estimated gain: 5-10% NPS. This is called on every non-pruned move.

  2. Avoid IsLegal make/unmake for en passant

  Currently IsLegal falls through to full make/unmake for en passant moves because of the tricky discovered check case. But en passant is rare
  (~0.1% of moves). This is already well-handled — the cost is negligible. Skip this.

  3. Incremental givesCheck in MakeMove

  A more ambitious version of #1: have MakeMove set a GivesCheck flag on the board. Since MakeMove already knows the piece, from, and to squares,
  it can check discovered + direct check cheaply as part of the move application. This eliminates the separate InCheck() call entirely.

  4. Node time check optimization

  Currently at search.go:656:
  time.Now()
  every 4096 nodes. time.Now() is a vDSO call (~20ns on Linux). Could use a monotonic counter or nanotime directly, but honestly 4096-node batches
   make this negligible. Not worth it.

  5. Move generation isn't allocating in the hot path

  The explorer flagged GenerateAllMoves() as allocating, but that function isn't used in search. The MovePicker uses
  GenerateCapturesAppend/GenerateQuietsAppend/GenerateEvasionsAppend which append to pre-allocated slices (mp.moves). The pickers are
  pre-allocated in info.pickers[ply] and reuse their slices. This is already well-optimized.

  My recommendation

  Do #1 — precomputed check squares. It's the clearest win:
  - Compute checkSquaresBB[6] once per node (6 piece types) using the enemy king square
  - Replace givesCheck := b.InCheck() with givesCheck := isCheckMove(piece, from, to, checkSquares, discoverers)
  - Eliminates a full IsAttacked() per searched move

  This is a well-understood optimization that most strong engines implement. Want me to plan it out?
